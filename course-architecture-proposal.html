Where the “Domain” tier should live in the data model

Recommended · low effort
A · Add a domain field
One enum/text column on the existing courses (now “Units”). Zero structural change, instantly groupable in the admin. Best first step.
Cleanest long-term
B · domains table
Real domains table + rename courses→units with an FK. Proper hierarchy, orderable, icon/colour per domain. More migration work.
Most flexible
C · Tags
Many-to-many tags so a unit can sit in several views (subject + path + exam-blueprint). Powerful but heavier UI.

Recommendation: ship A now to unblock the admin grouping, evolve to B when you want a first-class hierarchy.

Admin / course-creator experience

  • Drill-down tree: Domain → Unit → Module accordions with live counts at every level (modules · lessons · quiz · % content complete).
  • Filter bar: by Domain, by status (draft / published / empty), and a global search across unit & module titles.
  • Each Domain gets an icon + colour — reuse the existing icon/color fields so the dashboard is scannable at a glance.
  • “Empty” surfacing: a content-gap view that lists the ~half of units with 0 lessons, one click to Enhance.

What happens to “difficulty”

  • Drop Beginner/Intermediate/Advanced as a course-level construct (it was really just sequence).
  • Keep the learning arc as a recommended order number, not a gate — learners still get Foundations → Capstone.
  • Retain the existing per-question quiz_questions.difficulty for assessment analytics & adaptive practice. You lose the tiers, not the granular data.
